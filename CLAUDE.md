# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A Telegram bot (`src/bot.py`, built on `python-telegram-bot`) that proxies user messages to Gemini/Groq/OpenRouter LLMs through a small Orchestrator → Agent architecture, branded "Nela AI". Five internal modes (`general`, `math`, `fitness`, `writing`, `code`), but the product surfaces only one on the bottom keyboard (`general` — see `VISIBLE_MODES` in `src/i18n.py`); the rest still work if a user is switched into them, but the default "smart" general mode is expected to handle math/code/writing **and nutrition** without the user picking a mode. Nutrition is no longer a separate mode: the `general` system prompt auto-detects a food photo or meal description and responds as a nutritionist (КБЖУ table + a trailing `[NUTRITION_DATA]` marker), and any reply carrying that marker is logged to the diary. The keyboard is just `⚙️ Settings`.

## Commands

- Install deps: `pip install -r requirements.txt`
- Run the bot: `python -m src.bot` (entry point lives at `src/bot.py`; Dockerfile's `CMD` matches this)
- Run tests: `python -m unittest discover -s tests -p "test_*.py"`
- Run a single test file: `python -m unittest tests.test_llm_router`
- Run a single test case: `python -m unittest tests.test_llm_router.TestLLMRouter.test_gemini_key_cooldown_is_scoped_per_model`
- Run the flakiness loop (20x test runs): `python tests/run_checks.py`
- Run the formatting/latency benchmark: `python tests/run_benchmark.py`
- Lint: `flake8` (config in `.flake8`: max-line-length 120, ignores E402/W503/W291)

## Architecture

Request flow for a text message: `handlers/messages.py` → `Orchestrator.route_and_process` (`src/orchestrator.py`), which asks the embedding `Router` (`src/router.py`) which specialist should answer → that domain's `GenericAgent.process` (`src/agents/`) → `ask_llm` / `answer_with_web_tool` (`src/llm.py`) → `sender.send_response` (`src/sender.py`).

- **`src/i18n.py`** — single source of truth for the mode registry (`_MODES`/`MODE_KEYS`/`VISIBLE_MODES`/`DEFAULT_MODE`) and all user-facing strings (`STRINGS`, keyed by string name then `ru`/`en`). Adding a mode means one entry in `_MODES` plus a matching system prompt in `src/llm.py`'s `SYSTEM_PROMPTS`; adding a UI string means one entry in `STRINGS`.
- **`src/orchestrator.py`** — `Orchestrator` builds one `GenericAgent` for every routable domain (`general` + `MODE_KEYS` + the router's domains, so `nutrition` gets its own agent even though it isn't a keyboard mode) and dispatches by what the `Router` (see below) returns for the message. When the router is disabled (no embedding backend available — e.g. CI, or a config/network failure) it falls back to the caller's stored mode, so behaviour degrades to exactly what it was before routing existed. `GenericAgent` always parses the trailing `[NUTRITION_DATA]` marker and logs macros to the diary when present — the model only emits it after actually computing a meal, so non-food turns log nothing. (There's no separate `NutritionAgent` anymore; the photo path in `src/handlers/photos.py` does the same parse-and-log unconditionally.) Note: `"nutrition"` is gone from the keyboard mode *registry* (`_MODES`) but is a live routing domain and still has its `SYSTEM_PROMPTS` entry (`src/llm.py`) — `inline.py`'s `food` quick-lookup also calls `ask_llm(mode="nutrition")` directly, so don't delete that prompt assuming it's dead.
- **`src/router.py`** — the embedding intent router that decides WHICH specialist answers, before any answer is generated. It's a nearest-neighbour classifier over a FROZEN embedding model (no training): each specialist domain (`nutrition`, `math`, `fitness`, `writing`, `code`) has example phrases in `DOMAIN_EXAMPLES`; a message is embedded and routed to the domain whose examples are most similar, or to `general` (the fallback catch-all) if nothing clears `config.ROUTER_THRESHOLD`. Two pluggable backends behind one interface (`config.ROUTER_BACKEND`): **`gemini`** (default) — hosted multilingual embeddings via the Google API (a small network call; the bot is Russian-first and this is the only option that handles Russian well at zero RAM); **`local`** — a `model2vec` static model (`config.ROUTER_MODEL_PATH`) that runs on-CPU with no network, kept available for hosts that can bake in a good small model (the small English `potion` models route Russian poorly, and the multilingual one is 512MB = the whole free-tier RAM, which is why `gemini` is the default). Routing math that matters: **mean-centering** (subtract the global mean of the example vectors — embedding spaces are anisotropic, so without it cosine is ~0.9 for any pair and useless) then **max cosine over a domain's examples** (k-NN, not a blurred centroid). Everything is **fail-open**: if the backend can't init or a query errors, `route()` returns `general`, so the router can never break a reply. It logs every decision with its score, which is how you tune `ROUTER_THRESHOLD` from real traffic. Routing latency: ~0.05ms for `local`, ~80-150ms for `gemini`.
- **`src/agents/base.py`** — `BaseAgent` ABC requiring `process(user_input, history, user_settings, user_id) -> AgentResult`, where `AgentResult = (text, model, prompt_tokens, completion_tokens, latency)`. `text` is `None` if the whole LLM fallback chain failed, so the caller can show a clean error instead of logging an empty turn. This tuple carries real telemetry all the way to `log_usage_stats` — there's no placeholder/dummy data in the handler path.
- **`src/llm.py`** — All actual LLM I/O. Key behaviors to know before touching this file:
  - Fallback chain: direct Gemini API first (`KeyPool` tracks cooldowns per `(key, model)` pair, not just per key, since a key exhausting its daily quota on one Gemini model still has quota left on others), then Groq, then OpenRouter (free models first, then paid as a last resort). For a given model, every active key is tried (in random order) before the chain moves on to the next model.
  - Direct Gemini's `direct_models` list is ordered by real observed free-tier daily quota, not by model quality/recency — `gemini-3.1-flash-lite` leads (500 req/day per key on the AI Studio console; with N configured keys that's 500×N req/day before any other model is even tried), because the 2.0 tier once assumed to have the highest quota was observed in production hitting daily-quota 429s on every key first. If a model is reordered again, it should be based on the AI Studio console's actual per-model quota, not a recency/quality assumption.
  - Per-call HTTP timeout is deliberately short (`_FAST_TIMEOUT` = 10s, 5s connect) so one overloaded/slow model can't stall the whole fallback chain — a single call has been observed hanging 15s+ before erroring. `_THINKING_TIMEOUT` (25s) is the explicit exception, passed only for math mode and the `gemini-2.5-pro` deep fallback, since both deliberately leave Gemini's internal "thinking" on and can legitimately run long (see `THINKING_CONTROL_MODELS`/`THINKING_MODES`).
  - `_LLM_SEMAPHORE` (`config.LLM_MAX_CONCURRENCY`, default 6) bounds concurrent outbound calls so a burst of users can't trip rate limits for everyone at once.
  - `KeyPool.fail_key` differentiates failure types with different cooldown scopes/durations: invalid key (global to the key, 600s), daily quota exhaustion detected via `"perday"` in the error text (per-model, 21600s/6h), per-minute 429 (per-model, 60s), other 403/500/503 (per-model, 300s), and a generic exception/timeout (global to the key, 300s).
  - `is_response_complete()` heuristically detects truncated output (unbalanced ``` ``` ``` / `**`, dangling table rows) and is used to reject a model's response and fall through to the next model/key — this is a real control-flow signal, not just logging.
  - History is trimmed locally and cheaply via `trim_history` (pure truncation, no extra LLM call) rather than summarized — there is no LLM-based auto-summarization step.
  - `estimate_tokens` is a char-count heuristic (Cyrillic ~2 chars/token, other ~4 chars/token), used only as a fallback when the API doesn't return real usage metadata.
  - `src/handlers/admin.py` reaches into this module's internals directly (`key_pool`, `disabled_models`) to power `/admin`'s key-pool view and the `/disable_model`/`/enable_model` runtime kill switch — there's no abstraction layer between them, so renaming these breaks the admin panel too.
- **`src/database.py`** — async SQLite (`aiosqlite`) with WAL mode and FK constraints. `init_db()` is defensive: it inspects `PRAGMA table_info`/`PRAGMA foreign_key_list` at startup and migrates older schemas in place (renaming tables, recreating with FKs, copying data, backfilling new columns) rather than assuming a fixed schema — relevant if you add a column or table, follow the same migrate-in-`init_db` pattern, and don't assume any single column (e.g. `created_at`) is present on every legacy schema you migrate from. `DB_PATH` defaults to an absolute path on the dev machine but is overridden via the `DB_PATH` env var in Docker (`/data/bot.db`).
  - **Pseudonymous chat content (privacy):** the `messages` table does NOT store a raw `user_id` — it stores `conv_id`, a salted SHA-256 hash of the user id (`_conv_id()`), with no FK back to `users`, so chat content can't be casually attributed to a person by reading the DB (this is the `/privacy` promise). `log_message`/`get_chat_history`/`clear_chat_history`/`delete_last_exchange`/`get_all_chat_history` still take a real `user_id` and hash it internally — callers are unchanged. The hash salt is `config.PRIVACY_SALT` and MUST stay stable: changing it orphans all existing history (lookups would use a different hash). It's pseudonymity, not absolute anonymity (salt + a known user_id can recompute the hash). The shared `_migrate_messages_pseudonymous()` helper (run from both the SQLite `init_db` and the Turso schema init) anonymizes any legacy `user_id` rows in place on first startup. The `users` table keeps full identity plus a lifetime `message_count` (incremented per `'user'`-role message in `log_message`, surfaced as `messages_sent` in `get_user_activity_summary`); `stats`/`nutrition_log` still keep `user_id` (they're metrics, not message content).
- **`src/web_search.py`** — web search (Tavily) exposed as a tool the model calls, not a separate mode. `run_search(query)` enforces the daily budget, hits Tavily `basic` search, and returns grounding text + source URLs (or `None` — fail-open — when the key is missing, the budget is spent, or there are no results). `sources_footer(sources)` renders the tiny `(источник: [domain](url))` line (top 1-2). The decision to search lives in `llm.answer_with_web_tool` (see below), not here. Budget: a bot-wide per-day counter in the `search_counter` table (`database.get_today_search_count`/`increment_search_count`), capped by `config.TAVILY_DAILY_LIMIT` (default 50). Env: `TAVILY_API_KEY`, `TAVILY_DAILY_LIMIT`.
- **`src/llm.py` → `answer_with_web_tool`** — the general-mode answer path, using Gemini **function calling** so one model call both decides whether to search AND answers: plain chat costs a single call, only an actual search adds a second (the model emits a `web_search` tool call → we run `web_search.run_search` → feed the result back as a `functionResponse` → the model answers). The tool is only attached when `TAVILY_API_KEY` is set and the daily budget isn't spent (otherwise the model just answers from memory). On ANY failure it returns `text=None`, and `GenericAgent` falls back to the plain `ask_llm` chain (no search) — search never blocks a reply. `GenericAgent` appends `sources_footer` to a web-grounded answer. Only `general` uses this path; other modes go straight to `ask_llm`. The Gemini function-calling REST shape (request `tools`/`functionDeclarations`, response `functionCall` part, `functionResponse` sent back with `role: "user"`) was verified against the live API. `_compose_system_prompt` is shared by `ask_llm` and `answer_with_web_tool` so both behave the same on language/length.
- **`src/sender.py`** — send path with fallbacks. The DEFAULT is `sendRichMessage` (gated by `config.USE_RICH_MESSAGE`, default `true`; not yet in `python-telegram-bot`, so called directly via `httpx`): it sends the model's RAW Markdown and **up-to-date Telegram renders Markdown, LaTeX (`$...$`/`$$...$$`) and tables natively** — so on the rich path `to_telegram_html` is NOT involved. If `USE_RICH_MESSAGE` is off or the call fails, it falls back to `sendMessage` with HTML from `src/utils.to_telegram_html` (a hand-rolled Markdown/LaTeX→HTML converter; there LaTeX becomes monospace `<code>`, not real math), then to plain text. **Consequence for prompts:** the rich path renders a formula only when it's properly delimited, so the model must wrap every formula in `$...$`/`$$...$$` and never emit a bare `\frac`/`\nabla` — otherwise BOTH paths show raw LaTeX (this was a real bug: a math question rendered as `\nabla f = \left(\frac{...}{...}\right)` text). That rule lives in `_FORMATTING_PHILOSOPHY` in `src/llm.py` (appended to every mode's prompt), which matters because math questions reach the model through `general` (the only surfaced mode), not the `math` prompt.
- **`src/handlers/`** — thin Telegram-event glue. `message_handler` (`messages.py`) is the main text path; its `process_text_message` helper is reused by `voice.py` (which first transcribes the Opus/OGG via Gemini native audio, then feeds the transcript through the identical pipeline, so voice behaves exactly like typed text). `photos.py` runs vision and does the same `[NUTRITION_DATA]` parse-and-log as `GenericAgent`. `commands.py` holds `/start /mode /today /week /stats /clear /feedback /privacy /help` plus `build_main_keyboard` — which is JUST the `⚙️ Settings` button (Stats and New-chat/clear are buttons *inside* the Settings panel in `settings.py`). `reminders.py` adds `/remind HH:MM text` (stored as UTC) and a `check_reminders` job the bot runs every 60s (registered in `bot.py` via `job_queue.run_repeating`). `admin.py` and `inline.py` round it out. Bottom-keyboard button text is resolved via `src/i18n.resolve_button`, which maps either-language labels back to a `("settings"|"stats"|"clear", None)` (or `("mode", key)`) action, so buttons survive a UI-language switch mid-session.
- System prompts and user-facing strings are Russian-first; per-user `language` setting (`ru`/`en`) is appended as an instruction suffix to the system prompt rather than swapping prompt language wholesale.

## CI / deployment workflow

- **`.github/workflows/ci.yml`** runs `flake8 src/` + the full `unittest` suite on every push to `main` and every PR targeting `main` (Python 3.12, dummy `TELEGRAM_BOT_TOKEN`/`GROQ_API_KEY` env so `config.py` import-validation passes; tests mock LLM I/O so no real keys are needed). Lint is scoped to `src/` on purpose — the `tests/run_*.py` helper scripts have many pre-existing style violations and aren't worth gating on. The intended flow is **Variant A**: work on a branch → open a PR → CI must be green → merge to `main`; Render then auto-deploys `main`, so only tested code reaches prod. Don't commit straight to `main` (it bypasses the gate). **A merge does NOT guarantee a live deploy** — a Render deploy can finish `update_failed` (build or startup error) and keep the previous version serving, so after merging verify via the Render MCP that the newest deploy reaches `status: live` and the app logs show a clean start, not just that CI was green.

## Dev tooling

- **`mcp_server/turso_mcp.py`** + **`.mcp.json`** — a project-local, READ-ONLY MCP server that gives Claude Code a window into the Turso production DB for debugging (conversations, per-request stats, nutrition diary), since Turso is otherwise a blind spot. It reads `TURSO_DATABASE_URL`/`TURSO_AUTH_TOKEN`/`PRIVACY_SALT` from `.env`, only runs `SELECT`, and hashes `user_id`→`conv_id` exactly like `src.database._conv_id` (so the `conversation(user_id)` tool only matches if the salt equals production's). Its dependency (`mcp`) lives in `requirements-dev.txt`, NOT the bot's Docker image. Claude Code loads MCP servers at startup, so it only becomes available after a restart.
